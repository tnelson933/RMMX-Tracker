Master Replit Build Document
Rocky Mountain ATV/MC Branded RFID Motorcycle Race Tracking Platform
Product Summary
Build a web and mobile app combo for motorcycle race clubs and consumers to manage race registration, RFID-based check-in, live timing/results, lap times, scoring, series points, reporting, and public race standings.
The platform has two major user experiences:
1.	Club/Organizer Portal
•	private login
•	create and manage events
•	configure registration options dynamically
•	check riders in on race day
•	assign/link RFID transponders to riders
•	manage heats, qualifiers, moto formats, lineups
•	publish results
•	manage series points
•	export reports and print results
2.	Public Consumer App / Web View
•	no login required for basic access
•	view race results
•	sort races by state and event name
•	view live results and standings
•	view series leaderboards
•	mobile-friendly experience
This should be built as a scalable SaaS-style race management system with a future path for RFID hardware integration, timing system integration, and optional payment/checkout features.

STEP 1 — Foundation, Architecture, and Branding Layer
text
Create the foundation for a Rocky Mountain ATV/MC branded RFID motorcycle race tracking platform with a web app and mobile-friendly app-like experience.

Important: build the branding system as a theme layer that can be swapped or updated later. Use placeholder branding assets if real Rocky Mountain ATV/MC brand assets are not available or licensed. Structure the app so logos/colors/fonts can be changed centrally.

Build the app with these core experiences:
- Club/organizer login portal
- Public results portal with no login required
- Responsive UI for desktop, tablet, and mobile
- Race management, registration, check-in, RFID linking, live results, series points, and reporting

Architecture requirements:
- Use a modular full-stack structure
- Separate public pages, authenticated club pages, admin services, and shared UI components
- Prepare the app for database-backed registration, scoring, and results
- Build with future RFID hardware integration in mind
- Make the project scalable for multi-club, multi-event, multi-state use

Initial pages/routes:
- Public landing page
- Public results page
- Public event leaderboard page
- Club login page
- Club dashboard
- Event management pages
- Registration management pages
- Check-in pages
- Race day management pages
- Results and reports pages

Core UI requirements:
- Clean motorsports-inspired design
- Rocky Mountain ATV/MC-style visual hierarchy, but with swappable theme assets
- Strong desktop admin UI
- Mobile-friendly public results UI
- Fast search and filtering by state and event name

Set up the base database schema planning for:
- Clubs
- Users
- Events
- Riders
- Registrations
- RFID tags/transponder numbers
- Check-ins
- Moto/heats/qualifiers
- Results
- Lap times
- Series points
- Reports/export records

Start by creating the project structure, app shell, authentication framework, and theme system.

- Super Admin

- Public Visitor

- Secure login/logout

Club management features:

- Multiple organizer/staff users per club
- Assign roles to club staff
- Configure club branding/theme overrides if allowed

Security requirements:
- Protect all management pages behind login
- Keep public results pages open
- Ensure data isolation between clubs
- Add session persistence
- Add password reset support

Database models to implement:
- User
- Club
- ClubMembership
- Role
- Permission

Deliverable for this step:

STEP 3 — Event Creation and Dynamic Registration Builder
text
Build the event creation and rider registration system for club organizers.

Organizers must be able to create registration links for events that riders can use to register online.

Requirements:
- Create events by club
- Event details:
  - event name
  - date
  - location
  - state
  - track name
  - race classes
  - registration open/close dates
  - pricing
  - payment enabled/disabled
  - required rider fields
  - optional rider fields
- Dynamic registration builder

Dynamic registration options:
- Clubs can choose which fields are required or optional
- Clubs can enable or disable payment
- Clubs can choose class selections
- Clubs can define age, skill, bike class, and other race-specific custom fields
- Support flexible registration configurations so different race organizations can use different formats

Public rider registration flow:
- Rider opens event registration link
- Enters rider information
- Selects class(es)
- Pays if enabled
- Receives registration confirmation
- Registration is stored for check-in and race day management

Organizer tools:
- View all registrations
- Edit/void registrations if necessary
- Export rider lists
- See payment status
- See class breakdowns
- Generate registration links

Database models to implement or extend:
- Event
- EventRegistrationConfig
- RegistrationFieldConfig
- Rider
- RiderRegistration
- RegistrationClassSelection
- PaymentRecord

Deliverable:
- Clubs can create flexible event registration pages
- Riders can register through public links
- Payment can be enabled or disabled by event

STEP 4 — Day-of-Race Check-In, Rider Linking, and RFID Assignment
text
Build the race-day check-in system and RFID transponder linking workflow.

Requirements:
- Fast check-in interface for staff/organizers
- Search rider by name, bib, class, or registration status
- Show who has checked in and who has not shown up
- Allow RFID transponder numbers to be linked to rider profiles
- Allow staff to assign or update RFID numbers at check-in
- If a rider has not previously registered in the system, allow staff to create a quick rider record and link RFID manually

Check-in workflow:
- View all riders registered for the event
- Mark riders as checked in
- Assign RFID/transponder number
- Confirm bike/class selection
- Record arrival time
- Mark no-shows automatically after cutoff if needed

RFID requirements:
- Store RFID number/transponder ID per rider and per event if needed
- Support one-to-one mapping and event-specific overrides
- Make RFID assignment available to timing/results systems
- Validate that RFID-linked riders can be matched to results later

UI requirements:
- Extremely fast staff-facing check-in screen
- Large search bar
- Filter by class, checked-in status, RFID linked status
- Easy mobile/tablet use in race-day environments

Database models:
- CheckIn
- RFIDTag
- RiderRFIDLink
- EventAttendanceStatus
- QuickCreateRider

Deliverable:
- Organizers can check riders in quickly
- RFID is linked to each rider
- The system knows who is present and who is missing
- Race results can accurately match riders to transponders

Suggested Technical Stack for Replit
A strong stack for this project would be:
•	Frontend: Next.js or React
•	Backend: Node.js / Express or Next.js API routes
•	Database: PostgreSQL
•	ORM: Prisma
•	Auth: custom auth or NextAuth
•	Styling: Tailwind CSS
•	PDF/Export: server-side generation tools
•	PWA: optional service worker support
•	Real-time updates: WebSockets or polling for live results later
